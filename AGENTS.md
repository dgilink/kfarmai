# AGENTS.md - kFarmAI 7일 집중 업데이트 작업 지침

본 문서는 kFarmAI 공모전 발표 전 7일 동안 GitHub 저장소에서 작업할 때 사용할 에이전트/개발자 공통 작업 지침입니다. 목표는 현재 오류 없는 kfarmai.com 상태를 유지하면서, 발표 시점에는 “실제 작동형 모바일 웹 시제품”으로 보이도록 완성도를 끌어올리는 것입니다.

## 0. 최우선 목표

kFarmAI는 단순 데모가 아니라 다음 흐름이 실제 화면에서 이어지는 서비스로 보여야 합니다.

```text
검색/질문/사진·문장 입력
→ AI 참고 진단
→ 공공데이터 확인
→ 관련 질문·커뮤니티 축적
→ 날씨·시세·지도·가이드·계산기 등 다음 행동 참고
```

핵심 메시지는 항상 유지합니다.

> AI가 원인 후보를 정리하고, 공공데이터가 안전한 확인 기준을 제공하며, 커뮤니티가 해결 경험을 축적한다.

## 3. Git 안전 규칙

현재 kfarmai.com은 오류 없이 운영 중인 상태를 전제로 합니다. 모든 작업은 되돌릴 수 있어야 합니다.

### 3.1 작업 시작 전 반드시 실행

```bash
git status -sb
git branch --show-current
git log --oneline -5
```

확인할 것:

- 현재 브랜치가 무엇인지
- 수정 중인 파일이 이미 있는지
- 이전 미커밋 변경이 있는지
- 본인 작업과 무관한 파일이 있는지

### 3.2 절대 금지

```bash
git add .
git reset --hard   # 승인 없이 금지
git clean -fd      # 승인 없이 금지
npm audit fix --force
npm update
```

### 3.3 선택 add 원칙

항상 수정한 파일만 직접 지정합니다.

```bash
git add path/to/file1 path/to/file2
git diff --cached --stat
git diff --cached
```

### 3.4 작업 브랜치 권장

```bash
git checkout -b contest/2026-07-polish
```

이미 작업 브랜치가 있다면 그대로 사용합니다. main에서 직접 대규모 수정하지 않습니다.

### 3.5 안전 태그/백업

작업 전 안전 지점을 남깁니다.

```bash
git tag safety-baseline-YYYYMMDD-HHMM
```

또는 브랜치로 보관합니다.

```bash
git branch backup/before-contest-polish-YYYYMMDD
```

### 3.6 커밋 단위

하루 한 번 큰 커밋이 아니라, 기능 단위로 작게 커밋합니다.

예시:

```text
feat: improve home search result actions
feat: add public-info cards after AI reference diagnosis
feat: add related questions by crop symptom tags
fix: align pesticide safety wording
chore: update contest smoke test checklist
```

### 3.7 롤백 방법

아직 commit 전이면:

```bash
git restore path/to/file
```

commit 후, 아직 push 전이면:

```bash
git reset --soft HEAD~1
```

push 후 문제가 생기면 reset보다 revert를 우선합니다.

```bash
git revert <commit-hash>
```

배포 직전에는 반드시 현재 정상 commit hash를 기록합니다.

```bash
git rev-parse --short HEAD
```

## 4. 배포 안전 규칙

### 4.1 배포 전 체크

- 모바일 홈 정상 진입
- 검색 정상 작동
- AI 참고 진단 페이지 진입
- 커뮤니티 질문 페이지 진입

- 계산기 결과 표시
- 콘솔 에러 확인

### 4.2 API 오류 대비

공공데이터 API, 시세 API, 날씨 API는 실패할 수 있습니다. 모든 외부 데이터는 fallback 화면이 있어야 합니다.

예시 문구:

```text
현재 데이터를 불러오지 못했습니다. 잠시 후 다시 확인하거나 공식 공공데이터 제공처를 확인해주세요.
```

### 4.3 기능 플래그 권장

불안정한 신규 기능은 임시 플래그로 제어합니다.

```js
const FEATURE_RELATED_QUESTIONS = true;
const FEATURE_PUBLIC_INFO_MATCHING = true;
const FEATURE_SAME_SYMPTOM_REACTION = true;
const FEATURE_HELPFUL_REACTION = true;
const FEATURE_TRENDING_QUESTIONS = true;
const FEATURE_SMARTFARM_CHECKLIST = false;
```

발표 전 문제가 생기면 플래그를 false로 바꾸어 즉시 숨길 수 있게 합니다.

## 5. 데이터·AI 안전 규칙

### 5.1 AI 응답 기본 구조

AI 결과는 항상 다음 구조를 따릅니다.

```json
{
  "mode": "AI 참고 진단",
  "crop": "고추",
  "symptoms": ["잎 말림", "생육 저하"],
  "causeCandidates": [
    { "name": "해충 가능성", "strength": "중간", "reason": "잎 말림과 생육 저하가 함께 나타남" }
  ],
  "checkPoints": ["잎 뒷면 확인", "최근 고온 여부 확인"],
  "publicDataNeeds": ["농약안전사용지침", "병해충 정보", "농업날씨"],
  "safetyNotice": "이 결과는 AI 참고 진단이며 최종 처방이 아닙니다."
}
```

### 5.2 AI 프롬프트 안전 지침

AI에게 반드시 지시합니다.

```text
절대 특정 농약을 추천하지 말 것.
절대 최종 처방으로 말하지 말 것.
절대 구매를 유도하지 말 것.
원인 후보, 확인 포인트, AI 추정 강도, 공공데이터 확인 항목만 정리할 것.
농약 사용 전 등록 작물, 병해충, 사용시기, 희석배수, 안전사용기준 확인을 안내할 것.
```

### 5.3 개인정보·사진

- 사진 업로드 기능은 발표 전 저장 여부를 명확히 확인합니다.
- 사용자 사진을 저장한다면 동의 문구 필요.
- 저장하지 않는다면 “분석용으로만 사용” 등 현재 동작에 맞는 안내 필요.
- API 키는 절대 프론트엔드 코드에 노출하지 않습니다.

### 5.4 커뮤니티 반응·검색 안전 지침

커뮤니티 반응 기능은 SNS식 “좋아요 경쟁”이 아니라, 비슷한 증상과 유용한 참고자료를 찾기 위한 보조 기능으로 설계합니다.

우선 표현:

```text
나도 같은 증상
도움 됐어요
비슷한 질문 보기
지금 많이 보는 질문
최근 많이 공감한 증상
```

주의사항:

- “지역별 병해충 발생 예측”, “정확한 유행 정보”처럼 단정하지 않습니다.
- 반응 수는 진단 근거가 아니라 사용자 관심·경험 참고 신호로만 표시합니다.
- 데이터가 부족하면 기본 추천 질문과 공공자료 기반 참고 콘텐츠를 함께 보여줍니다.
- 로그인 없이 누르는 반응은 localStorage anonymous_id 등으로 중복 클릭을 최소화하되, 완전한 통계라고 표현하지 않습니다.
- 발표 전 오류가 생기면 FEATURE_SAME_SYMPTOM_REACTION, FEATURE_HELPFUL_REACTION, FEATURE_TRENDING_QUESTIONS 플래그로 즉시 숨길 수 있어야 합니다.

## 6. 7일 작업 계획

## Day 1 - 홈 첫 화면·통합검색 고도화

### 목표

홈 검색을 단순 검색창이 아니라 문제해결 진입점으로 만듭니다. 초기 방문자가 “무엇을 검색해야 할지” 몰라도 지금 많이 보는 질문과 같은 증상이 많은 질문을 통해 자연스럽게 들어오도록 설계합니다.

### 작업

1. 홈 검색 결과를 유형별로 구분합니다.
   - AI 참고 진단
   - 관련 질문
   - 공공자료 기반 참고 콘텐츠
   - 공공정보 확인
   - 도구: 날씨, 시세, 계산기
   - 지역 정보: 지도, 기관 링크
2. 검색 결과 상단에 빠른 이동 카드를 추가합니다.
   - AI 참고 진단으로 확인하기
   - 관련 질문 보기
   - 공공정보 확인
   - 도구·지도 보기
3. 검색창 아래에 초기 방문자용 탐색 영역을 추가합니다.
   - 지금 많이 보는 질문
   - 나도 같은 증상이 많은 질문
   - 이번 주 많이 확인할 문제
4. 검색 결과가 없을 때 대체 행동을 제시합니다.
   - AI 참고 진단으로 확인
   - 커뮤니티에 질문하기
   - 관련 공공정보 보기
5. 검색어 예시를 홈에 표시합니다.
   - 고추 잎 말림
   - 토마토 잎 반점
   - 몬스테라 잎 노래짐
   - 상토 곰팡이
   - 액비 희석

### 추천 홈 구성

```text
검색창
AI 참고 진단 시작 카드
지금 많이 보는 질문
나도 같은 증상이 많아요
이번 주 많이 확인할 문제
```

### 데이터 구조 예시

```json
{
  "title": "고추 잎이 말릴 때 확인할 점",
  "type": "question",
  "crop": "고추",
  "symptoms": ["잎말림", "생육저하"],
  "tags": ["고추", "잎말림", "해충", "고온"],
  "view_count": 12,
  "same_symptom_count": 5,
  "helpful_count": 3,
  "created_at": "2026-07-01",
  "url": "/questions/pepper-leaf-curl.html"
}
```

### 정렬 기준

```text
지금 많이 보는 질문 = view_count 높은 순 + 최근성
나도 같은 증상 = same_symptom_count 높은 순
최근 질문 = created_at 최신순
```

초기 데이터가 적으면 기본 시연용 질문과 공공자료 기반 참고 콘텐츠를 함께 보여줍니다. 단, “인기 통계”가 아니라 “시제품 추천 질문” 또는 “많이 확인하는 질문”처럼 안전하게 표현합니다.

### 완료 기준

- 모바일 홈에서 검색 후 유형별 결과가 보인다.
- 홈 검색 아래에 “지금 많이 보는 질문”과 “나도 같은 증상” 섹션이 보인다.
- 검색 결과 없음 화면에서 AI 참고 진단 또는 질문하기로 이동할 수 있다.
- 기존 검색 기능이 깨지지 않는다.

### 테스트 검색어

```text
고추 잎 말림
토마토 흰가루
딸기 잎끝 마름
몬스테라 노란잎
상토 곰팡이
```

## Day 2 - AI 참고 진단 결과 화면 안전화

### 목표

AI가 답을 단정하지 않고, 원인 후보와 확인 포인트를 정리하는 구조를 명확히 보여줍니다.

### 작업

1. 모든 화면의 “AI 진단” 표현을 “AI 참고 진단”으로 통일합니다.
2. 결과 카드를 다음 항목으로 구성합니다.
   - 원인 후보
   - 확인 포인트
   - AI 추정 강도
   - 공공데이터 확인 필요
   - 안전사용기준 확인
3. 안전 문구를 결과 상단 또는 하단에 고정합니다.

안전 문구:

```text
이 결과는 AI 참고 진단이며 최종 처방이 아닙니다. 농약 사용 전 등록 작물, 병해충, 사용시기, 희석배수, 안전사용기준을 반드시 확인하세요.
```

4. AI 결과에서 태그를 추출하거나 저장할 수 있는 구조를 만듭니다.
   - crop
   - symptoms
   - causeCandidates
   - publicDataNeeds

### 완료 기준

- 결과 화면에서 “AI 참고 진단”이 명확히 보인다.
- 특정 농약 추천, 구매 유도 표현이 없다.
- 공공정보 확인으로 이어질 태그가 저장된다.

## Day 3 - 공공정보 확인 레이어 구현

### 목표

공모전용 문구가 아니라 실제 서비스 문구로 “공공정보 확인” 흐름을 만듭니다.

### 작업

1. AI 참고 진단 결과 하단에 “확인할 공공정보” 섹션을 추가합니다.
2. 다음 카드를 표시합니다.
   - 농약안전사용기준 확인
   - 병해충 정보 / NCPMS 확인
   - 농업날씨 확인
   - 농산물 시세 참고
   - 지역 정보 확인
3. 내부 공공정보 인덱스를 만듭니다.

```json
[
  {
    "title": "농약안전사용기준 확인",
    "category": "safety",
    "tags": ["농약", "방제", "병해충", "안전사용기준"],
    "description": "등록 작물, 병해충, 사용시기, 희석배수 확인",
    "url": "/public/pesticide-safety.html"
  }
]
```

4. AI 결과 태그와 공공정보 인덱스를 연결합니다.
5. API가 실패해도 카드가 보이도록 fallback 구조를 둡니다.

### 표현 원칙

- “공공데이터 공모전 서비스”라고 쓰지 않습니다.
- 사용자 화면에는 “공공정보 확인”, “안전사용기준 확인”, “확인할 공공정보”라고 표현합니다.

### 완료 기준

- AI 참고 진단 결과에서 공공정보 확인 카드로 이동한다.
- 농약안전사용기준, 병해충 정보, 날씨, 지도, 시세가 확인 경로로 보인다.
- API 실패 시에도 안내 문구가 뜬다.

## Day 4 - 커뮤니티·관련 질문·반응 기능 강화

### 목표

커뮤니티를 단순 게시판이 아니라 질문, 비슷한 증상, 도움 반응, 공공정보 확인이 쌓이는 구조로 만듭니다. 농업·식물 문제에서는 “다른 사람도 같은 증상을 겪는지”가 중요한 정보이므로, 검색과 커뮤니티에 반응 데이터를 연결합니다.

### 작업

1. AI 참고 진단 결과 하단에 버튼을 추가합니다.
   - 공공정보 확인
   - 비슷한 질문 보기
   - 커뮤니티에 질문하기
2. 질문 카드 또는 참고 콘텐츠 카드에 반응 버튼을 추가합니다.
   - 나도 같은 증상
   - 도움 됐어요
   - 공공정보 확인
3. 질문 데이터에 태그와 카운트를 붙입니다.
   - crop
   - symptoms
   - causeCandidates
   - cultivationType
   - view_count
   - same_symptom_count
   - helpful_count
4. 비슷한 질문 추천은 태그 기반으로 구현합니다.

간단 점수 방식:

```text
같은 작물명 +3점
같은 증상 +3점
같은 원인 후보 +2점
같은 계절/지역 +1점
```

5. “나도 같은 증상” 반응은 검색/홈 추천에 반영합니다.
   - same_symptom_count가 높은 질문을 “나도 같은 증상이 많아요” 섹션에 표시
   - view_count가 높은 질문을 “지금 많이 보는 질문” 섹션에 표시
6. 커뮤니티 질문 작성 화면에 AI 참고 진단 요약을 자동 입력합니다.

### 질문 작성 기본 항목

필수:

```text
작물/식물명
증상
사진 또는 설명
```

선택:

```text
지역
재배유형
최근 날씨
사용 이력
발생 시기
```

### 반응 기능 구현 안전안

1차 구현은 너무 복잡하게 만들지 않습니다.

```text
나도 같은 증상: same_symptom_count 증가
도움 됐어요: helpful_count 증가
```

중복 클릭 방지는 가능한 범위에서 처리합니다.

- 로그인 사용자는 user_id 기준
- 비로그인 사용자는 localStorage anonymous_id 기준
- 완전한 통계로 표현하지 않고 “반응 수”, “같은 증상 표시 수”로 표현

### 버튼 문구

```text
나도 같은 증상 12
도움 됐어요 8
비슷한 질문 보기
공공정보 확인
```

### 홈/검색 반영

홈 검색 아래에는 다음 섹션을 우선 표시합니다.

```text
지금 많이 보는 질문
나도 같은 증상이 많아요
이번 주 많이 확인할 문제
```

검색 결과에서도 같은 구조를 사용합니다. 예를 들어 “고추”를 검색하면 AI 참고 진단, 많이 본 질문, 나도 같은 증상, 공공정보 확인, 농업날씨, 계산기를 함께 보여줍니다.

### 완료 기준

- AI 결과에서 비슷한 질문으로 이동한다.
- 질문 카드에 “나도 같은 증상”, “도움 됐어요” 버튼이 보인다.
- 같은 증상 반응 수가 홈 또는 검색 추천 영역에 반영된다.
- 커뮤니티 질문 작성 시 작물명·증상·AI 참고 요약이 자동 반영된다.
- “전문가 답변”, “발생 예측”, “정확한 통계”처럼 책임이 큰 표현은 사용하지 않는다.

## Day 5 - 농자재·관리정보 허브 개편

### 목표

농자재 탭을 판매 목록이 아니라 작물관리·홈가드닝·스마트팜까지 연결되는 관리정보 허브로 확장합니다.

### 추천 카테고리

```text
작물보호
토양·배지
비료·액비
종자·육묘
관수·시설환경
홈가드닝
스마트팜·시설재배
보조사업
관련기관
```

### 작업

1. 농자재 탭 명칭과 설명을 안전하게 정리합니다.

추천 문구:

```text
농자재 구매가 아니라, 작물 관리와 공공정보 확인을 돕는 참고 정보입니다.
```

2. 4개 제조사 정보는 “초기 참고자료”로 표시합니다.

```text
주요 농자재 제조사 정보는 시제품에 일부 반영했으며, 향후 카테고리별로 보완할 계획입니다.
```

3. 보조사업 정보 구조를 만듭니다.

```json
{
  "title": "상토 지원사업",
  "region": "전남 순천",
  "category": "상토",
  "source": "지자체 공고",
  "summary": "상토 구입비 지원 관련 공고 확인",
  "url": "공식 공고 링크",
  "status": "수집·정리 중"
}
```

4. 보조사업 안내 문구를 넣습니다.

```text
보조사업 정보는 공식 공고 확인을 돕는 참고자료입니다. 신청 전 지자체 공고와 담당기관 확인이 필요합니다.
```

5. 관련기관 링크를 문제해결 흐름에 연결합니다.
   - AI 참고 진단 결과 → 공공정보 확인 → 관련기관
   - 보조사업 → 지자체/농업기관 링크

### 완료 기준

- 농자재 탭이 판매 중심으로 보이지 않는다.
- 홈가드닝, 작물관리, 스마트팜·시설재배 확장성이 보인다.
- 보조사업과 기관 링크가 단순 링크 모음이 아니라 다음 확인 경로로 보인다.

## Day 6 - 농업날씨·재배달력·재배가이드·시세·계산기 고도화

### 목표

AI 참고 진단 이후 사용자가 실제 판단에 참고할 수 있는 도구를 정리합니다.

### 6.1 농업날씨

추가 항목:

```text
기온
강수확률
강수량
습도
풍속
```

해석 문구는 단정하지 않습니다.

```text
비가 오거나 습도가 높을 경우 방제 작업 전후 주의가 필요할 수 있습니다.
```

### 6.2 재배달력

작물 선택형으로 정리합니다.

```text
고추 / 토마토 / 오이 / 딸기 / 벼 / 상추
```

월별 표시:

```text
파종
정식
생육관리
병해충 주의
수확
```

### 6.3 재배가이드

문제 중심 카드형으로 구성합니다.

```text
물관리
웃거름
병해충 주의
고온기 관리
수확 전 확인
```

각 카드에서 관련 기능으로 이동합니다.

```text
병해충 주의 → 병해충 정보 확인
고온기 관리 → 농업날씨 확인
웃거름 → 비료/액비 계산기
```

### 6.4 농산물 시세

API 승인/오류에 대비합니다.

```text
API 정상 → 최신 시세 표시
API 오류 → 최근 저장 데이터 또는 안내 화면 표시
```

안전 문구:

```text
농산물 시세는 판매·중개 목적이 아니라 시장 흐름을 참고하기 위한 공공 시세 정보입니다.
```

### 6.5 계산기

현재 계산기:

```text
상토 계산기
액비 희석 계산기
농약 희석 계산기
```

농약 계산기는 안전 문구 필수:

```text
계산 결과는 참고용입니다. 농약 사용 전 등록 작물, 병해충, 사용시기, 희석배수, 안전사용기준을 반드시 확인하세요.
```

추가 후보는 하나만 검토합니다.

```text
재배면적 단위 변환기
또는 관수량 참고 계산기
```

### 완료 기준

- 날씨, 달력, 가이드, 시세, 계산기가 서로 연결된다.
- 시세 API 오류가 나도 빈 화면이 나오지 않는다.
- 계산기 결과가 구매·처방처럼 보이지 않는다.

## 9. 최종 원칙
